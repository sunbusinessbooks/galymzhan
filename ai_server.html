Рекомендуемый формат

Один master HTML как единый источник правды. В нём должны быть:

  • полный текст архитектуры;
  • встроенные SVG-схемы;
  • иконки и пиктограммы без внешних ссылок;
  • версии сервисов и моделей;
  • блоки для печати в PDF.
Плюс такого формата: он открывается в любом браузере, печатается в PDF, отправляется как один файл и не разваливается из-за отсутствующих картинок.

Что ещё держать рядом

  • /docs — этот master document;
  • /infra — docker-compose, k8s, ansible, terraform;
  • /rag — ingestion, chunking, retrieval, reranking;
  • /services — API и оркестратор;
  • /benchmarks — тесты качества и latency;
  • /ops — логи, мониторинг, runbooks.

1. Концепция системы

Сервер на 4× A100 — это не просто хост для моделей. Это локальный AI Core Node: единый узел, который принимает запрос, направляет его в нужный сервис, объединяет результаты и возвращает контролируемый ответ.

2. Железо: производственная конфигурация

Рекомендуемая база

  • 4× NVIDIA A100 40/80 GB
  • AMD EPYC или Intel Xeon серверного класса
  • 256–512 GB ECC RAM
  • 2–4× NVMe enterprise для hot layer
  • отдельный слой SSD/HDD под архив и датасеты
  • 10/25 GbE, лучше с запасом под кластеризацию
  • redundant PSU и продуманное охлаждение

Почему один узел

  • минимальная задержка между сервисами;
  • простая отладка и контроль;
  • меньше сетевых бутылочных горлышек;
  • удобное закрепление GPU за сервисами;
  • проще вводить observability и ограничения.

3. Жёсткое разделение ролей по GPU

GPU1 — основная LLM

Модели: Qwen2.5, Llama 3.x, Mixtral, DeepSeek. Задачи: чат, reasoning, summarization, tool routing, final answer synthesis.

GPU2 — multimodal / vision

Модели: Qwen2-VL, LLaVA, Florence-2. Задачи: скриншоты, визуальные документы, интерфейсы, image-to-text, VQA.

GPU3 — embeddings + retrieval

Модели: BGE, E5, GTE, rerankers. Задачи: эмбеддинги, поиск, гибридное извлечение, reranking, сборка контекста.

GPU4 — agents / code / reserve

Модели: DeepSeek-Coder, Code Llama, planner/executor, batch jobs, тест новых пайплайнов, резерв под пиковую нагрузку.

4. Софт-стек: конкретные службы

Инфраструктура

Ubuntu Server, NVIDIA drivers, CUDA, Docker, Docker Compose, Proxmox VE или bare metal. При росте — Kubernetes.

Model serving

vLLM для LLM, TGI как альтернатива, SGLang для некоторых сценариев, Ollama — только для простых локальных экспериментов, не как основа production.

OCR / Parsing

PaddleOCR, Surya, DocTR, Unstructured, PyMuPDF, pdfplumber, Tesseract как fallback, layoutparser для структуры страниц.

RAG / Data

Qdrant как основной vector DB, PostgreSQL для метаданных, Redis для кэшей и очередей, MinIO для объектного слоя.

Оркестрация

FastAPI, Celery или Dramatiq, LangGraph или LlamaIndex workflows, Nginx/Traefik как edge-прослойка.

Наблюдаемость

Prometheus, Grafana, Loki, OpenTelemetry, Sentry, health checks, трассировка latency по этапам.

5. Правильный RAG: не декоративный, а продовый

Основная ошибка слабых систем — воспринимать RAG как «PDF загрузили → chunk сделали → embedding сделали → готово». Это даёт шум, потерю структуры, плохие ответы и нестабильность качества.

Плохой RAG

  • нарезка только по символам;
  • игнор таблиц, колонок, шапок, футеров;
  • одна retrieval-стратегия;
  • нет reranking;
  • нет метаданных, source-trace и версионирования.

Правильный RAG

  • ingestion с типизацией документа;
  • OCR и layout-aware parsing;
  • semantic chunking и parent-child chunks;
  • hybrid retrieval: BM25 + dense vectors;
  • reranking перед генерацией;
  • filters по источнику, дате, типу, отделу, языку;
  • контроль контекста и цитируемость.

Рекомендуемые модели RAG

  • Embeddings: BAAI/bge-large, bge-m3, intfloat/e5-large-v2, Alibaba-NLP/gte-large
  • Rerank: BAAI/bge-reranker-large, jina-reranker, Cohere Rerank если есть внешний сервис
  • Generation: Qwen2.5-Instruct, Llama 3.x Instruct, Mixtral

Обязательные практики

  • раздельные пайплайны для договоров, сканов, таблиц, регламентов;
  • хранение оригинального источника и координат chunk;
  • версионирование индекса;
  • benchmarks на recall@k, MRR, latency и groundedness.

8. Агентный слой

Агенты нужны не для красивого слова, а когда задача многошаговая: сначала разобрать запрос, потом получить данные, потом проверить правила, затем сформировать результат и отправить его дальше.

Planner

Разбивает цель на шаги.

Executor

Вызывает сервисы и инструменты.

Verifier

Проверяет groundedness, формат и полноту.

9. Наблюдаемость и эксплуатация

Что измерять обязательно

  • latency по этапам;
  • GPU utilization и memory pressure;
  • ошибки OCR и parsing;
  • retrieval recall, rerank lift;
  • quality drift по доменам;
  • стоимость запроса и размер контекста.

Службы

  • Prometheus + Grafana;
  • Loki / Elasticsearch для логов;
  • OpenTelemetry для трассировки;
  • Sentry для ошибок приложения;
  • alerting в Telegram / Slack / почту.

10. План внедрения без компромиссов

Этап 1 — инфраструктура

Поднять сервер, драйверы, контейнерную среду, модельный runtime, мониторинг, хранилища и API gateway.

Этап 2 — data plane

Сделать ingestion, OCR, parsing, индексацию, Qdrant, метаданные, версионирование и quality checks.

Этап 3 — serving

Развести роли по GPU, подключить LLM, vision, rerank, кэши, роутинг запросов и auth.

Этап 4 — eval + hardening

Подготовить эталонные наборы, промерить latency, recall, hallucination rate, добавить политики ответа и резервирование.

Правильная упаковка — это не один красивый текст. Это связка: master document + инфраструктурный код + тестовые наборы + наблюдаемость + контроль качества. Тогда систему можно реально внедрять, а не просто показывать.

11. Набор конкретных моделей и служб для старта

Базовый production stack

  • LLM serving: vLLM
  • Primary LLM: Qwen2.5-Instruct или Llama 3.x Instruct
  • Vision: Qwen2-VL / Florence-2
  • OCR: PaddleOCR + PyMuPDF + Unstructured
  • Embeddings: bge-m3 / e5-large-v2
  • Rerank: bge-reranker-large
  • Vector DB: Qdrant
  • API: FastAPI
  • Queue: Redis + Celery

Необходимые инженерные слои

  • RBAC и auth;
  • audit logs;
  • quota и rate limits;
  • prompt/version registry;
  • dataset registry;
  • offline eval и regression checks.

Для применения без компромиссов держи один self-contained master HTML как главный документ, а рядом — инфраструктурный репозиторий с кодом развёртывания, ingestion/RAG пайплайнами, наблюдаемостью и evaluation. Тогда это становится не текстом о системе, а самой операционной основой проекта.
